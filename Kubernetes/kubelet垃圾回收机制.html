    kubelet垃圾回收机制

    missionOne 2022/11/28 Kubernetes

    kubernetes上的垃圾回收，指的是回收节点上无用的容器和镜像，这个工作是由kubelet定期完成的

    # 垃圾回收周期

    kubelet每分钟会查询清理一次容器，每五分钟查询清理一次镜像。在kubelet刚启动时并不会立即进行GC，即第一次进行容器回收为kubelet启动一分钟后，第一次进行镜像回收为kubelet启动五分钟后。

    不推荐使用其它管理工具或手工进行容器和镜像的清理，因为kubelet需要通过容器来判断pod的运行状态，如果使用其它方式清除容器有可能影响kubelet的正常工作

    # 垃圾回收范围

    • 镜像的回收针对node结点上由docker管理的所有镜像，无论该镜像是否是在创建pod时pull的；
    • 容器的回收策略只应用于通过kubelet管理的容器；

    kubelet是通过集成的cadvisor进行镜像的回收

    # 垃圾回收策略配置

    kubelet支持如下的一些参数来配置垃圾回收策略：

    # 镜像回收

    • --image-gc-high-threshold：当存储镜像的磁盘使用率高于这个值的时候启动垃圾回收，删除最近最久未使用（LRU，Least Recently Used）的镜像；这个值是百分比；
    • --image-gc-low-threshold：当启动垃圾回收清理磁盘后，直到磁盘使用率降到这个值或无镜像可以删除为止；这个值是百分比；

    --image-gc-high-threshold的默认值为90，--image-gc-low-threshold的默认值为80；

    当某个镜像重新pull或启动某个pod用到该镜像时，该镜像的最近使用时间会被更新

    这里注意一点，docker默认使用/var/lib/docker存储镜像，当docker使用自定义的磁盘空间如/data时，kubelet默认垃圾回收机制不生效，需要在kubelet启动时增加--docker-root=来指定docker数据目录才行。

    # 容器回收

    • --minimum-container-ttl-duration：从容器停止运行时经过这个值的时间后，将该容器标记为已过期可回收，这里只是标记并未直接回收，值可以使用单位后缀，如h表示小时，m表示分钟，s表示秒；
    • --maximum-dead-containers-per-container：每个pod最多可保留这个数值个停止的容器；
    • --maximum-dead-containers：整个节点最多可保留这个值个停止的容器；

    --minimum-container-ttl-duration的默认值为1m，--maximum-dead-containers-per-container的默认值为2，--maximum-dead-containers默认值为100；

    --maximum-dead-containers的优先级是高于--maximum-dead-containers-per-container的；

    容器回收时是按照时间排序先回收创建时间最早的容器；

    # 关闭容器垃圾回收

    如果需要关闭容器的垃圾回收策略，可以将--minimum-container-ttl-duration设为0，表示无限制；--maximum-dead-containers-per-container和--maximum-dead-containers设为负数。

    # 垃圾回收流程

    1. 遍历所有pod，使其满足--maximum-dead-containers-per-container；

    2. 经过上一步后如果不满足--maximum-dead-containers，计算值X=（--maximum-dead-containers）/（pod总数），再遍历所有pod，使其满足已停止运行的容器集个数不大于X且至少为1；

    3. 经过以上两步后如果还不满足--maximum-dead-containers，则对所有已停止的容器排序，优先删除创建时间最早的容器直到满足--maximum-dead-containers为止;

    namespace和cgroup pod优雅关闭重启
